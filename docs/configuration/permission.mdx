Ante has a permission system that controls whether tool calls are auto-approved, require user confirmation, or are denied entirely.

## Policies

A **policy** sets the baseline behavior for a session:

| Policy | Behavior |
|---|---|
| `default` | Each tool call is evaluated against permission rules. If no rule matches, the user is prompted for approval. |
| `yolo` | All tool calls are auto-approved — no prompts, no rules evaluation. |

Set the policy in `~/.ante/settings.json`:

```json
{
  "policy": "default"
}
```

Or override per-session with the `--yolo` flag:

```bash
ante --yolo "run cargo clippy and fix all warnings"
```

<Warning>
In headless mode, `yolo` policy is implied — all tools are auto-approved since there is no interactive prompt.
</Warning>

## Decisions

When a tool call is evaluated, the result is one of three decisions:

| Decision | Effect |
|---|---|
| **Allow** | Tool executes immediately, no prompt |
| **Ask** | User is prompted to approve or deny |
| **Deny** | Tool call is blocked |

## Rules

Permission rules pair a **decision** with a **tool matcher**. Rules are evaluated in order — **first match wins**.

```
Rules:  [ Ask(Bash), Allow(Write), Deny(Bash(rm *)) ]
                ↓
Tool call: Bash({ "command": "ls" })
                ↓
Match: Ask(Bash)  ← first rule that matches wins
                ↓
Result: Ask (prompt user)
```

If no rule matches, the default decision is **Ask**.

### Default rules

When a session starts, Ante builds rules from tool metadata. Tools marked `requires_approval` get an `Ask` rule; others get `Allow`:

| Tool | Default decision |
|---|---|
| `Bash` | Ask |
| `Write` | Ask |
| `Edit` | Ask |
| `Read` | Allow |
| `Glob` | Allow |
| `Grep` | Allow |
| `BashOutput` | Allow |
| `KillShell` | Allow |
| `Task` | Allow |
| `TodoWrite` | Allow |
| `WebFetch` | Allow |
| `WebSearch` | Allow |

### Session-level overrides

When you approve a tool during a session (e.g. "always allow Bash"), Ante inserts an `Allow` rule at the **front** of the rule list. Because evaluation is first-match-wins, this effectively overrides the default `Ask` for that tool for the remainder of the session.

## Tool matchers

A tool matcher identifies which tool calls a rule applies to. There are two forms:

### Simple matcher

Matches **any** invocation of a tool by name (case-insensitive):

```
Bash
Write
Task
```

### Specifier matcher

Matches a tool **only when** its primary argument matches a glob pattern:

```
Bash(cargo test *)
Bash(npm run *)
Task(explore)
```

The syntax is `ToolName(glob-pattern)`. The glob is matched against the tool's primary argument:

| Tool | Primary argument |
|---|---|
| `Bash` | `command` |
| `Task` | `subagent_type` |

For other tools, only the simple (name-only) matcher is supported.

### Glob syntax

Specifiers use standard glob patterns:

| Pattern | Matches |
|---|---|
| `cargo test *` | `cargo test --all`, `cargo test my_mod` |
| `npm run *` | `npm run build`, `npm run test` |
| `safe-cmd:*` | `safe-cmd:foo`, `safe-cmd:bar` (but **not** `safe-cmd --flag`) |
| `exp*` | `explore`, `experiment` |

<Note>
The glob `safe-cmd:*` matches `safe-cmd:` followed by anything. It does **not** match `safe-cmd --flag` because `*` does not match across the `:` vs space boundary in the same way. Use `safe-cmd *` (with a space) to match space-separated arguments.
</Note>

## Tool filtering

Separate from permissions, you can control which tools are **available** in a session. Filtered-out tools are not presented to the agent at all.

```bash
# Only allow these tools (whitelist)
ante --allowed-tools Read Glob Grep "analyze the code"

# Remove these tools (blacklist)
ante --disallowed-tools Bash Write "read-only analysis"
```

Tool filtering supports the same ToolMatcher syntax:

```bash
# Allow Bash but only for specific commands
ante --allowed-tools Read "Bash(cargo test *)" "Bash(cargo clippy *)" "analyze and test"
```

<Tip>
Tool filtering and permissions are independent. Filtering controls what tools **exist** in the session; permissions control whether existing tools **require approval**.
</Tip>

## Mode-specific behavior

### TUI mode

In the TUI, tools that require approval show an interactive prompt. You can approve or deny each call, or choose to always allow a tool for the session.

### Headless mode

Headless mode implies `yolo` policy — all tool calls are auto-approved. Use `--allowed-tools` or `--disallowed-tools` to restrict what the agent can do:

```bash
# Restrict to read-only tools in CI
ante --allowed-tools Read Glob Grep "summarize the codebase"
```
